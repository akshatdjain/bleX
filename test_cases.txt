BleGod - Professional System Test Protocol (v2.1)
===================================================

This protocol defines 15 "Heavy Testing" cases designed to stress the provisioning, MQTT bridge, and registration architecture of the BleGod system.

---

### CATEGORY 1: DISCRETE PROVISIONING (HEAVY)

TEST CASE 1: Individual MQTT-Only Push (Minimal Payload)
- Action: On the Scanners tab, find a connected scanner. Tap the "MQTT" action button only.
- Logic: The App sends ONLY `mqtt_host` and `mqtt_port`.
- Practical Expectation: The scanner's `provisioner_service.py` receives the JSON, saves it to `~/mqtt_config.json`, and returns 200 OK. The scanner MUST NOT reset its Wi-Fi connection or drop from the network.

TEST CASE 2: Individual Full Wi-Fi Push (Complete Payload)
- Action: Enter valid Wi-Fi SSID/PSK in "Wi-Fi Creds". Tap "Wi-Fi" action button on a scanner card.
- Logic: The App sends `ssid`, `psk`, `mqtt_host`, and `mqtt_port`.
- Practical Expectation: The scanner deletes its current connection, creates a new `nmcli` connection, and switches from the 'setup' hotspot to the new site Wi-Fi. It MUST also save the MQTT host.

TEST CASE 3: Bulk "Push MQTT to All" Stress Test
- Action: Discover 5+ scanners via the tablet hotspot. Tap "Push MQTT to All".
- Heavy Testing Aspect: Verify the tablet's IO scope handles 5 parallel HTTP POST requests without ANR or dropping any payload.
- Practical Expectation: All 5 scanners should simultaneously report "200 OK" and update their internal broker settings without rebooting.

TEST CASE 4: Bulk "Push Wi-Fi to All" Network Transition
- Action: With multiple scanners discovered, tap "Push Wi-Fi to All".
- Heavy Testing Aspect: This triggers multiple scanners to leave the 'setup' AP at once.
- Practical Expectation: Monitors the tablet's connection pool. Scanners should gracefully exit the 'setup' network. Check the site router to ensure all scanners appear there within 30 seconds.

---

### CATEGORY 2: MQTT ARCHITECTURE & DYNAMICS

TEST CASE 5: Dynamic Host Resolution (Hotspot Re-entry)
- Action: Change the tablet's hotspot name or reconnect to a different router (changing Tablet IP). Open the Configurator.
- Check: Verify the "Tablet IP (MQTT Host)" display updates to the new IP.
- Action: Push "MQTT" to a scanner.
- Practical Expectation: The scanner MUST update its target broker IP to the new Tablet IP instantly.

TEST CASE 6: MQTT Payload Loopback & UI Latency
- Action: Place a beacon next to a provisioned scanner.
- Loop: Beacon -> Scanner -> MQTT -> Tablet Broker -> App UI.
- Heavy Testing Aspect: Check the "RSSI" value in the UI. Move the beacon.
- Practical Expectation: The UI should update within <500ms. If Kalman filtering is enabled, the value should smooth out without jumping.

TEST CASE 7: Broker Failure & Reconnect Watchdog
- Action: On the Tablet, toggle "Enable Embedded Broker" to OFF while scanners are active. Wait 1 minute, then toggle ON.
- Practical Expectation: Scanners should log "Connect failed, retrying...". Once the broker is ON, scanners MUST auto-reconnect using the `connect_mqtt_forever` loops without manual intervention.

---

### CATEGORY 3: REGISTRATION & PERSISTENCE

TEST CASE 8: Multi-Identity Conflict (MAC Spoofing Check)
- Action: Try to register a scanner with a name, then try to register another scanner using the SAME MAC address but a different name.
- Practical Expectation: The Backend API (FastAPI) should handle the conflict. Depending on implementation, it should either update the existing record or return a 400 error. The UI must display the result badge correctly.

TEST CASE 9: Tablet Hub Identity Preservation
- Action: Register "This Tablet" in the Scanners tab. Uninstall the app, reinstall it, and check the "This Tablet" card.
- Practical Expectation: Because the app uses ANDROID_ID, it should generate the same MAC-like ID. The status should reflect the registration status on the server, not just local storage.

TEST CASE 10: Scanner Name Overflow & UI Clipping
- Action: Register a scanner with an extremely long name (50+ characters).
- Heavy Testing Aspect: Check the ScannerCard layout.
- Practical Expectation: The name should use `TextOverflow.Ellipsis` and not break the 3-button layout or push the buttons off-screen.

---

### CATEGORY 4: SECURITY & NETWORKING (TLS/SSL)

TEST CASE 11: TLS 1.3 Handshake Resilience
- Action: Enable TLS on the broker, upload a custom `ca.crt` to the app. Point a scanner to the TLS port.
- Practical Expectation: Verify `HostnameInsensitiveSocketFactory` allows the connection even if the certificate's CN doesn't match the raw IP (which is common in local IoT).

TEST CASE 12: Invalid Port & Connection Timeout
- Action: Manually change the MQTT port to a blocked port (e.g., 9999) in settings and push to a scanner.
- Practical Expectation: The App should report a "HTTP Timeout" or "Connection Refused" error badge on the card rather than hanging indefinitely.

---

### CATEGORY 5: AGING & LOAD TESTS

TEST CASE 13: 10-Scanner "Heartbeat" Flood
- Action: Connect 10 scanners to the tablet. Each scanner sends a heartbeat + beacon data every 1s.
- Heavy Testing Aspect: Monitor the Tablet's Moquette broker memory usage.
- Practical Expectation: The Hub must remain responsive, and the persistent notification must stay active without being killed by the OS (Low Memory Killer).

TEST CASE 14: Wi-Fi "Zombie" Recovery (Safety Fallback)
- Action: Provision a scanner with WRONG Wi-Fi credentials.
- Practical Expectation: The scanner's `check_zombie_fallback` (Zombie Protection) should detect the loss of connectivity and, after the threshold, return the Wi-Fi interface to the 'setup' hotspot so it can be re-provisioned.

TEST CASE 15: Configuration Change Survival (Rotation Stress)
- Action: While a "Push to All" action is in progress (progress bar visible), rotate the tablet 180 degrees.
- Heavy Testing Aspect: Kotlin `rememberCoroutineScope()` and `ViewModel` survival.
- Practical Expectation: The push operation MUST CONTINUE in the background, and the result dialog/toast MUST appear correctly in the new orientation without crashing the UI thread.
