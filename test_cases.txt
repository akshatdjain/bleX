BleGod - Comprehensive System Test Cases
========================================
Test Environment: Android Tablet/Smartphone (Target API 35 / Android 16 compatible)
Pre-requisites: APK installed, 1 or more BLE Scanners (e.g. ESP32-S3), 1 or more BLE Beacons to detect.

--- CATEGORY: PERMISSIONS & INITIALIZATION ---

Test Case 1: Fresh Install & Permission Flow
- Action: Install the app for the first time and launch it.
- Expected Output: The app should systematically ask for Bluetooth, Location (Fine/Coarse), Nearby Devices, and Notification permissions. Denying Location should show a rationale dialog explaining why BLE needs Location. The Scanning service should not start until permissions are granted.

Test Case 2: Tablet Reboot Auto-Start
- Action: Once the app is running and permissions are granted, completely reboot the Android device. Do not manually open the app.
- Expected Output: The BootReceiver should trigger automatically. After the device boots, the persistent "Scanner Status" notification should appear on the lock screen/notification panel, proving the scanning service started in the background automatically.

--- CATEGORY: HARDWARE TOGGLES & INTERRUPTIONS ---

Test Case 3: Bluetooth Toggle Mid-Scan
- Action: While the app is actively scanning and finding beacons, swipe down and turn Bluetooth entirely OFF from the Quick Settings menu. Wait 10 seconds, then turn it back ON.
- Expected Output: The app's watchdog should detect the BLE failure. The UI should show scanning halted. Once BT is flipped back on, the app should automatically recover, restart the BLE scanner, and resume updating the UI and MQTT broker without requiring an app restart.

Test Case 4: Wi-Fi Toggle & Offline Resilience
- Action: With the Local MQTT Broker enabled, publish beacons. Turn Wi-Fi completely OFF.
- Expected Output: The UI should warn that the network adapter is down. The Local MQTT Broker should remain running on the localhost interface `127.0.0.1`. The app should queue or discard remote MQTT payloads gracefully without crashing, and restore remote connection silently when Wi-Fi is restored.

--- CATEGORY: BACKGROUND & PERSISTENCE ---

Test Case 5: Persistent Notification Un-Swipeable Test
- Action: Lock the screen. Wake it up to view the lock screen. Try to swipe away the "Scanner Status" notification. Then unlock the phone, pull down the notification shade, and try to swipe it away again.
- Expected Output: The notification should be strictly un-swipeable and persistent (spoofing CATEGORY_CALL behavior). It should display as a single horizontal line "BLE: Active | MQTT: Up | Remote: Down/Up".

Test Case 6: App Killed via Recent Apps (Self-Healing)
- Action: Open the app, ensure scanning is active. Press the recent apps button and swipe the BleGod app completely away (force kill via UI). Lock the screen.
- Expected Output: The notification might briefly disappear. Within 5 to 10 seconds, the ServiceRestartReceiver MUST catch the death and automatically respawn the foreground service. The notification will reappear, and background scanning will resume without user intervention.

Test Case 7: Doze Mode & Battery Optimization
- Action: Start the scanner, lock the screen, unplug the tablet from power, and leave it sitting completely still on a desk for 3+ hours to force Android into Deep Doze mode.
- Expected Output: The WakeLock and Foreground Service should prevent the CPU from sleeping. Upon waking the device, the total beacons scanned counter should accurately reflect 3 hours of constant scanning, and the MQTT broker should have continuous logs. Check Android battery stats to ensure it shows foreground usage, not a crash.

--- CATEGORY: MQTT & NETWORK ---

Test Case 8: Embedded Broker Toggle
- Action: Go to Settings Tab. Toggle "Enable Embedded MQTT Broker" OFF. Look at the persistent notification. Then toggle it ON again.
- Expected Output: Turning it OFF should cleanly stop the Moquette broker and the notification should instantly update to "MQTT: Down". Turning it ON should restart it on port 1883 and update the notification to "MQTT: Up".

Test Case 9: Remote Broker Auto-Reconnect
- Action: Configure a remote MQTT broker with intentional invalid credentials, save. Then change to valid credentials.
- Expected Output: The MqttBridge should attempt to connect, fail, and log an error. Upon entering the correct credentials, the watchdog should automatically retry and successfully establish the bridge, pushing data to the remote server.

--- CATEGORY: PROVISIONING & SCANNERS TAB ---

Test Case 10: Tablet Self-Registration
- Action: Navigate to the Configurator -> Scanners Tab. Look at the top section labeled "This Tablet".
- Expected Output: The card should correctly display the Tablet's underlying hardware model and its Bluetooth MAC Address. Tapping "Register" should save it to the local SQLite database and change the status badge to "Registered".

Test Case 11: Network Scanner UDP Discovery
- Action: Power on an ESP32-S3 configured to broadcast UDP to the tablet's IP on port 8888. Navigate to the Scanners Tab.
- Expected Output: The UI should automatically discover the new scanner, listing its IP and MAC address dynamically in the "Network Scanners" list without needing a manual refresh.

Test Case 12: Pull-to-Refresh Scanners
- Action: While on the Scanners tab, wait for scanners to list. Physically power off an ESP32 scanner. Wait 20 seconds, then perform a swipe-down Pull-to-Refresh on the screen.
- Expected Output: The tablet should flush the stale 15-second cache. The powered-off scanner should disappear from the UI list cleanly.

Test Case 13: Push Wi-Fi Credentials Pixel-Style
- Action: Tap the "Wi-Fi Config" button next to a discovered Network Scanner. Enter an SSID and Password, then submit.
- Expected Output: A UDP payload containing `wifi_ssid`, `wifi_pass`, `mqtt_ip` (tablet's IP), and `mqtt_port` (1883) should be dispatched to the target scanner. Verify on the scanner's serial output that it received the credentials and connected to the tablet's hotspot.

--- CATEGORY: CONFIGURATOR & ZONES ---

Test Case 14: Zone Creation & DB Persistence
- Action: Go to the Configurator -> Zones Tab. Create 3 new zones (e.g., "Hallway", "Kitchen", "Lobby"). Force close the app, and reopen it.
- Expected Output: The zones must persist in the SQLite database and reload correctly when navigating back to the Zones tab.

Test Case 15: Assigning Beacons to Zones
- Action: In the Zones tab, drag/assign a newly discovered raw beacon MAC address to the "Kitchen" zone. Name it "Keys".
- Expected Output: The beacon should immediately disappear from the "Unassigned" pool. Future scans in the Scanner Tab should now resolve this MAC address to the friendly name "Keys - Kitchen".

Test Case 16: Cascading Zone Deletion
- Action: Delete the "Kitchen" zone created in Test 15. Accept the confirmation dialog.
- Expected Output: The zone is deleted. The beacon "Keys" should be automatically orphaned and thrown back into the "Unassigned" raw MAC address pool.

Test Case 17: Zone Chips Scroll & Layout
- Action: Create 15 different zones with long names.
- Expected Output: The zone selection row at the top of the Zones tab should gracefully horizontally scroll. Selected zones should highlight in the Material 3 primary color, matching the app's dynamic theme.

--- CATEGORY: EDGE CASES & STRESS TESTS ---

Test Case 18: High Volume Beacon Stress Test
- Action: Place the tablet in an extremely busy BLE environment (e.g., an office with 100+ BLE beacons/devices, or simulate it using a BLE spoofer). Let it run for 10 minutes.
- Expected Output: The app should not freeze (ANR). The RecyclerView in the Scanner tab should remain smooth (maintaining 90Hz/120Hz refresh rates) because the BLE parsing and DB sorting are happening asynchronously on Dispatchers.IO.

Test Case 19: Screen Rotation & Configuration Changes
- Action: While staring at the Scanners Tab with incoming live data, rapidly rotate the tablet from Portrait to Landscape and back 5 times.
- Expected Output: The UI state should survive the configuration changes cleanly without duplicating data, crashing, or restarting the background BLE Scanner Service.

Test Case 20: Long Duration Soak Test (24 Hours)
- Action: Plug the tablet into a charger, clear all app data, set it up, start scanning, and leave it running untouched for exactly 24 hours.
- Expected Output: 
   1) No OutOfMemory (OOM) crashes. 
   2) The local database sizes should remain manageable.
   3) The total scan count should be in the millions.
   4) CPU usage should have remained stable (no memory leaks).
   5) MQTT connection to remote (if set) should have auto-healed through any overnight ISP router drops.
